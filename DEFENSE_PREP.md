## 二、答辩话术

### 2.1 开场介绍（1-2分钟）

> 老师好，我的毕业设计题目是《基于 RAG 技术的智能知识问答系统》。
>
> 这是一个面向企业和个人用户的智能知识问答平台。用户可以直接上传各种格式的文档，系统会自动解析、分块、向量化并存储，然后通过 AI 模型自动匹配最相关文件，对用户的问题进行检索增强生成回答。
>
> 项目的核心技术亮点有六个：
> 1. **RAG 检索增强生成**：基于 BGE 中文 Embedding + ChromaDB 向量检索的精准问答，减少 AI 幻觉
> 2. **流式响应**：SSE 实时输出 + 超时保护，用户体验像 ChatGPT 一样流畅
> 3. **多轮对话**：支持上下文记忆，理解用户追问意图
> 4. **文件引用模式**：知识广场采用引用而非复制，节省存储空间，避免数据冗余
> 5. **Token 自动刷新**：JWT 双 Token 认证，Access Token 过期无感续期
> 6. **全站深色模式**：CSS 变量方案，所有页面完整适配

### 2.2 技术选型说明（1-2分钟）

> 技术栈方面，我选择的是：
>
> **后端**：Python + FastAPI + SQLAlchemy + MySQL
> - 选择 FastAPI 是因为它原生支持异步，性能优秀，且自带 API 文档
> - SQLAlchemy 是成熟的 ORM 框架，支持异步操作
>
> **向量存储**：ChromaDB
> - 轻量级向量数据库，几行代码即可集成，支持本地持久化
> - 基于 HNSW 算法，百万级向量毫秒级响应
>
> **LLM**：Ollama + Qwen2:7b
> - 本地部署，数据不离开本地，保护隐私
> - 零成本，无需 API 费用
>
> **前端**：Vue 3 + Element Plus + Pinia
> - 学习曲线平缓，中文生态好
> - Element Plus 组件丰富，开发效率高

> **1. 文件上传**

### 2.4 技术难点突破（2-3分钟）

> 项目中遇到的最大挑战是**向量检索与异步框架的兼容性问题**。
>
> **问题描述**：
> - ChromaDB 的查询是同步操作
> - FastAPI 是异步框架，直接调用会阻塞事件循环
> - 导致其他请求无法处理，系统性能下降
>
> **解决方案**：
> - 使用 `asyncio.to_thread()` 将同步操作放到线程池中执行
> - 这样既保证了检索效率，又不影响其他请求的处理
>
> ```python
> # 解决方案
> results = await asyncio.to_thread(vector_store.query, kb_id, question, 5)
> ```
>
> 另一个难点是**流式聊天的 Session 作用域问题**。
>
> **问题描述**：
> - StreamingResponse 在 endpoint 返回后才消费生成器
> - 此时 FastAPI 注入的 db session 已关闭
>
> **解决方案**：
> - 在生成器内部创建独立的 database session
> - 所有 DB 操作使用新 session，不依赖注入的 session

### 2.5 安全设计说明（1-2分钟）

> 在安全方面，我设计了多层防护机制：
>
> **认证安全**：
> - JWT 双 Token 认证：Access Token 60分钟 + Refresh Token 7天
> - Token 自动刷新：Access Token 过期时自动使用 Refresh Token 续期，用户体验无中断
> - Token 轮换机制：Refresh Token 使用后立即失效，防止重放攻击
> - 密码安全：PBKDF2 算法 + 随机盐值，100,000 次迭代
> - 时序攻击防护：使用 `hmac.compare_digest()` 常量时间比较
>
> **数据安全**：
> - XSS 防护：DOMPurify 消毒所有 HTML 输出
> - SQL 注入防护：SQLAlchemy ORM 参数化查询
> - 频率限制：基于 IP 的登录/注册频率限制
> - 文件上传验证：前后端双重类型白名单 + 大小限制（50MB）

## 三、常见问题与答案

### Q1: 为什么选择 RAG 而不是微调模型？

**答**：主要考虑三个因素：
1. **成本**：微调需要大量计算资源和标注数据，RAG 只需存储文档向量
2. **灵活性**：RAG 可以随时更新知识库，微调需要重新训练
3. **可解释性**：RAG 的每个回答都有来源引用，用户可以验证

### Q2: 向量检索的原理是什么？

**答**：向量检索基于 Embedding 技术。首先用 Embedding 模型将文本转换为高维向量（如 512 维），然后通过余弦相似度计算向量间的距离。ChromaDB 使用 HNSW 算法实现近似最近邻搜索，可以在百万级数据中毫秒级返回最相似的结果。

### Q3: 如何保证回答质量？

**答**：我从三个方面保证：
1. **检索质量**：设置相似度阈值（0.3），过滤不相关的结果
2. **Prompt 设计**：明确要求 LLM 只基于参考内容回答，不确定时说明
3. **来源引用**：每个回答标注参考来源，用户可以验证

### Q4: 流式响应是怎么实现的？

**答**：后端使用 FastAPI 的 StreamingResponse + async generator，每个 LLM 输出的 chunk 作为 SSE 事件发送。前端使用 fetch + ReadableStream 接收数据，通过 buffer 机制处理分片问题。这样用户可以实时看到 AI 生成的内容，体验像 ChatGPT 一样。

### Q5: 为什么选择 ChromaDB 而不是 Milvus？

**答**：主要考虑部署复杂度。ChromaDB 是轻量级的，几行代码即可集成，支持本地持久化，适合中小型项目。Milvus 是分布式向量数据库，适合大规模生产环境，但部署和运维成本更高。对于我的项目规模，ChromaDB 完全够用。

### Q6: 多轮对话是怎么实现的？

**答**：每次用户提问时，我会从数据库中加载最近 5 轮对话历史，格式化后注入到 Prompt 中。这样 LLM 可以理解上下文，支持追问和澄清。历史消息会截断到 500 字符，避免超过 LLM 的上下文窗口限制。

### Q7: 如何处理 LLM 超时？

**答**：使用 `asyncio.wait_for()` 设置 120 秒超时。如果超时，返回友好提示「AI 模型响应超时，请稍后重试」。同时在前端实现重试机制，用户可以重新发送问题。

### Q8: 密码存储安全吗？

**答**：使用 PBKDF2 算法 + 随机盐值存储密码，迭代 100,000 次。验证时使用 `hmac.compare_digest()` 进行常量时间比较，防止时序攻击。即使数据库泄露，攻击者也很难暴力破解密码。

### Q9: 项目有哪些可以优化的地方？

**答**：
1. **缓存**：可以添加 Redis 缓存热点知识库和设置项
2. **分页**：已经实现了分页，但前端可以添加无限滚动
3. **测试**：需要补充单元测试和集成测试
4. **部署**：可以容器化部署，使用 Docker Compose 编排

### Q10: 如果用户量增大，系统如何扩展？

**答**：
1. **水平扩展**：FastAPI 支持多 worker 部署，可以增加实例
2. **数据库**：MySQL 可以读写分离，ChromaDB 可以切换到 Milvus
3. **缓存**：引入 Redis 缓存热点数据
4. **LLM**：Ollama 可以部署多实例，或切换到云端 API
