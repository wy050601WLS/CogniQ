# 知识问答系统 - 功能扩展规划

## 项目现状

当前系统已完成核心功能开发，包括：
- 用户端：知识库管理、文档上传、RAG 问答、收藏、反馈
- 管理端：用户管理、知识库管理、数据统计、系统设置
- 后端：完整的 REST API、JWT 认证、流式响应

---

## 一、短期扩展（1-2周）

### 1.1 多轮对话记忆

**当前问题：** 对话是单轮独立的，每次问答不依赖历史上下文。

**实现方案：**
- 将最近 N 轮对话历史作为 context 传给 LLM
- 在 `chat_service.py` 中查询当前对话的最近消息
- 构建 messages 数组：`[历史问题1, 历史回答1, 历史问题2, ..., 当前问题]`
- 设置 token 上限，超出时截断早期对话

**预期效果：** 用户可以追问、澄清、深入探讨，问答质量显著提升。

---

### 1.2 对话导出

**当前问题：** 用户无法保存有价值的对话内容。

**实现方案：**
- 后端新增 `GET /api/conversations/{id}/export?format=pdf|md|txt`
- PDF 导出使用 `reportlab` 或 `weasyprint`
- Markdown 导出直接格式化消息列表 
- 前端在对话详情页添加"导出"按钮
 
**预期效果：** 用户可将问答结果保存为文档，便于分享和归档。Su

---

### 1.3 引用追溯

**当前问题：** 来源引用只显示片段，无法查看完整上下文。

**实现方案：**
- 在来源数据中存储文档 ID 和 chunk 位置
- 前端点击来源时跳转到知识库详情页
- 高亮显示引用的原始段落

**预期效果：** 用户可验证 AI 回答的准确性，增强信任感。

---

### 1.4 异步文档处理

**当前问题：** 大文件上传时服务器阻塞，用户体验差。

**实现方案：**
- 使用 `Celery` + `Redis` 实现异步任务队列
- 文档上传后返回任务 ID，前端轮询状态
- 处理完成后通过 WebSocket 或轮询通知用户
- 添加任务进度条显示

**预期效果：** 大文件不再阻塞，用户可继续其他操作。

---

## 二、中期扩展（1-2月）

### 2.1 知识图谱

**当前问题：** 知识是扁平的文档块，缺乏结构化展示。

**实现方案：**
- 使用 LLM 进行实体抽取和关系识别
- 存储实体-关系三元组到 Neo4j 或内存图结构
- 前端使用 D3.js 或 vis.js 绘制关系图谱
- 支持按实体搜索、路径查询

**预期效果：** 用户可直观看到概念之间的关联，发现隐含知识。

---

### 2.2 知识库版本控制

**当前问题：** 文档修改后无法回溯历史版本。

**实现方案：**
- 文档表增加 `version` 字段
- 每次修改创建新版本记录，保留旧版本
- 向量数据按版本隔离存储
- 前端提供版本历史列表和对比功能

**预期效果：** 支持文档回滚，误操作可恢复。

---

### 2.3 Webhook 通知

**当前问题：** 事件发生后用户无感知。

**实现方案：**
- 后端维护 Webhook 配置表
- 关键事件触发时调用用户配置的 URL
- 支持事件类型：文档处理完成、知识库更新、用户被禁用
- 提供 Webhook 测试和日志查看功能

**预期效果：** 支持自动化工作流集成，适合团队协作场景。

---

### 2.4 标签系统

**当前问题：** 知识库和文档缺乏分类能力。

**实现方案：**
- 新增 `tags` 表，支持多对多关联
- 知识库和文档都可以打标签
- 前端支持按标签筛选和搜索
- 支持用户自定义标签

**预期效果：** 用户可按主题、项目、场景分类管理知识。

---

### 2.5 对话分享

**当前问题：** 有价值的对话无法分享给他人。

**实现方案：**
- 生成带 token 的分享链接
- 链接有效期可配置（1天/7天/永久）
- 只读模式，无法编辑和追问
- 支持设置访问密码

**预期效果：** 知识可快速传播，适合团队知识共享。

---

## 三、长期扩展（3-6月）

### 3.1 多租户架构

**当前问题：** 所有用户共享数据库，无数据隔离。

**实现方案：**
- 引入 `tenant_id` 字段隔离数据
- 中间件自动注入租户上下文
- 向量库按租户分 collection
- 支持租户级别的配额和权限

**预期效果：** 可部署为企业 SaaS 产品，支持多组织使用。

---

### 3.2 RAG 策略可配置

**当前问题：** 检索参数固定，无法适配不同场景。

**实现方案：**
- 设置页面增加 RAG 参数配置：
  - 检索数量（top_k）
  - 相似度阈值
  - 重排序策略
  - 分块大小
- 不同知识库可独立配置
- 提供参数调优建议

**预期效果：** 用户可根据文档特点调优问答效果。

---

### 3.3 协作功能

**当前问题：** 知识库只能单人管理。

**实现方案：**
- 知识库成员角色：管理员、编辑者、查看者
- 协作编辑、评论、@通知
- 变更历史和操作日志
- 实时协同编辑（WebSocket）

**预期效果：** 支持团队协作，适合企业知识管理。

---

### 3.4 多模态支持

**当前问题：** 只支持文本类文档。

**实现方案：**
- PDF 图片使用 OCR 提取文字
- 表格使用 `camelot` 或 `tabula` 解析
- 图片使用多模态 LLM 生成描述
- 支持 PPT、Excel 格式

**预期效果：** 可处理更多类型的文档内容。

---

### 3.5 API 开放平台

**当前问题：** 系统只能通过界面使用。

**实现方案：**
- 提供完整的 RESTful API
- API Key 认证和限流
- 详细的 API 文档和 SDK
- 开发者门户

**预期效果：** 支持第三方集成，扩展生态系统。

## 五、实施建议

1. **优先级排序：** 多轮对话 > 对话导出 > 引用追溯 > 知识图谱
2. **渐进式开发：** 每次专注一个功能，验证后再做下一个
3. **用户反馈驱动：** 收集实际使用反馈，调整优先级
4. **技术验证先行：** 复杂功能先做 PoC，确认可行性
